BUG 2 — CURRENT CITY IS NOT APPLIED RELIABLY

Real-device behavior:

User says or types:
"Taikos 61"

The app sometimes resolves Taikos 61 in another city.

Expected:

If the user is physically in Klaipėda, the first search must be scoped to Klaipėda.

Required investigation:

Trace the actual source of locality:

- last known GPS coordinates
- location freshness timestamp
- reverse-geocoded locality
- subAdminArea fallback
- cached city
- whether city is null at resolution time

Use tag:
KentasLocationContext

Required fix:

Create one reliable current-location context source for destination resolution.

For every short street-number address:

1. Use a recent device location if available.
2. Reverse geocode coordinates to locality.
3. Accept locality or subAdminArea only if non-blank.
4. Cache the resolved city with coordinates and timestamp.
5. Reuse the cache only while reasonably fresh.
6. If location permission was granted after app startup, refresh location and city context.
7. If location is stale or unavailable, do not silently search another city.

Suggested freshness:
- location: up to 5 minutes
- city cache: up to 15 minutes, provided coordinates have not moved far

When city is known:

"Taikos 61"
must generate only city-scoped candidates first:

"Taikos 61, Klaipėda, Lithuania"
"Taikos pr. 61, Klaipėda, Lithuania"
"Taikos prospektas 61, Klaipėda, Lithuania"

Reject results from another city.

Do not fall back to an unscoped nationwide result without user confirmation.

If current city is unavailable, ask:

"Kuriame mieste yra šis adresas?"

Do not navigate to a guessed city.
