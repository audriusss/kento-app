TASK: Fix the real-device SpeechRecognizer loop. ERROR 11, stuck listening UI, silent failures, and missed commands must be eliminated before any new feature work.

REAL DEVICE SYMPTOMS

Confirmed on a Xiaomi phone:

- Voice destination resolution now works.
- The route starts successfully.
- Continuous mode appears active.
- UI shows:
  - "Pokalbis leidžiamas"
  - "Kentas klauso..."
- But after the first successful command:
  - Kentas sometimes stops reacting.
  - The UI may still say that it is listening even though recognition is dead.
  - Sometimes the UI shows:
    "Balso atpažinimo klaida (kodas: 11)"
  - Kentas does not speak any recovery message.
  - The recognizer may not recover automatically.
  - The user has to interact manually again.

This is a real-device failure, not a theoretical edge case.

PRIORITY

Do not add new features.
Do not expand waypoint work.
Do not redesign the app.
Do not change Google Navigator lifecycle.
Do not push automatically.

Fix only the voice recognition session lifecycle and recovery until it is stable.

ROOT GOAL

After the user presses the microphone once:

1. Kentas listens.
2. User speaks.
3. Kentas processes the command.
4. Kentas responds.
5. After TTS finishes, listening restarts exactly once.
6. This loop continues reliably.
7. Any recognizer error recovers predictably.
8. UI must never claim "Kentas klauso..." unless SpeechRecognizer is genuinely in an active listening session.

Ensure there is only one active recognition session at a time.

Implement a generation/session token.

Example:

Ignore stale callbacks from an older generation.

Before startListening():

- verify app is foreground

- verify TTS is not speaking
- verify no recognition session is active
- verify no restart is already pending
- verify recognizer instance exists and is healthy

Do not call startListening() twice for the same generation.

Do not allow overlapping recognizer sessions.

PHASE 5 — BOUNDED RETRY AND BACKOFF

Prevent rapid infinite retry loops.

Suggested retry policy:

- NO_MATCH: 500 ms
- SPEECH_TIMEOUT: 500 ms
- RECOGNIZER_BUSY: 800 ms and recreate after repeated occurrence
- CLIENT: recreate recognizer and retry after 1000 ms
- SERVER_DISCONNECTED / code 11 if applicable: recreate and retry after 1200 ms
- NETWORK: 1500–3000 ms
- TOO_MANY_REQUESTS: longer cooldown, e.g. 5000 ms

Maximum automatic consecutive recovery attempts:

3

After 3 consecutive hard failures:

- stop automatic retry temporarily
- keep continuous mode visibly enabled only if a manual retry button or toggle can recover
- show:
  "Balso atpažinimas laikinai neveikia. Paliesk mikrofoną, kad bandytum dar kartą."
- do not show "Kentas klauso..."

Reset retry count after a successful onResults or stable onReadyForSpeech session.

PHASE 6 — TTS AND RECOGNIZER COORDINATION

The recognizer must never listen while Kentas is speaking.

Requirements:

- set TTS speaking state synchronously before calling tts.speak()
- onStart confirms speaking
- onDone clears speaking
- onError clears speaking
- recognizer restart occurs only after onDone/onError
- add a short post-TTS delay, e.g. 500 ms
- cancel all older restart jobs before scheduling a new one

Critical rule:

There must be only one restart path.

Do not restart from multiple places such as:

- command handler exit
- TTS onDone
- recognizer onResults
- ViewModel coroutine completion

Choose one central coordinator.

If a command does not produce TTS:

- explicitly notify the coordinator that processing finished
- schedule one restart

If a command produces TTS:

- TTS onDone is the only restart trigger

PHASE 7 — REMOVE RACE CONDITIONS

Audit for these race conditions:

1. startListening called before old onError finishes
2. old onResults arrives after a new session starts
3. delayed restart job runs after the microphone was toggled off
4. TTS onDone from an old utterance restarts a new disabled session
5. lifecycle background event races with retry job
6. recognizer recreated while old callbacks still arrive
7. Compose recomposition creates duplicate callbacks
8. SpeechRecognizerManager and VoiceSessionController both independently restart listening
9. onResults and onError both trigger recovery for the same session
10. command processing finishes twice

Use generation IDs or atomic guards to make stale callbacks harmless.

PHASE 9 — USER FEEDBACK ON ERRORS

Do not make Kentas speak on every harmless timeout.

Rules:

NO_MATCH / SPEECH_TIMEOUT:
- silent automatic retry
- no TTS

Recoverable recognizer reset:
- UI message only:
  "Atkuriamas balso atpažinimas..."

Repeated hard failure:
- one concise TTS message:
  "Balso atpažinimas laikinai neveikia."

Permission error:
- TTS if possible:
  "Reikia mikrofono leidimo."
- show actionable UI

Avoid TTS loops where the error message itself causes another recognizer failure.

PHASE 10 — LIFECYCLE SAFETY

On app background:

- cancel active listening
- cancel pending restart
- mark session inactive
- keep or disable continuous mode according to current intended UX
- do not restart in background

On foreground:

- do not restart until lifecycle is fully resumed
- if continuous mode was intentionally kept enabled, restart once
- otherwise remain idle

Do not destroy Google Navigator.

Do not create duplicate SpeechRecognizer instances on recomposition.

SpeechRecognizer creation and destruction must occur on the main thread.
