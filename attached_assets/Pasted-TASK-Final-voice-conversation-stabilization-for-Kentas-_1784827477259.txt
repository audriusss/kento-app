PHASE 2 — MAKE THE RECOGNIZER WAIT LONGER

Configure SpeechRecognizer for natural Lithuanian speech.

Use these initial values unless device behavior proves they are unsupported:

EXTRA_PARTIAL_RESULTS = true

EXTRA_SPEECH_INPUT_MINIMUM_LENGTH_MILLIS:
1500–2000 ms

PHASE 3 — UTTERANCE AGGREGATION

Implement a small utterance aggregator.

Track:
- latest partial transcript
- latest final transcript
- time of latest meaningful transcript change
- recognizer generation/session ID
- whether speech has actually begun
- whether TTS is active
- whether the utterance was already delivered

Rules:

1. Partial results update the candidate transcript only.
2. Do not execute commands from partial results.
3. Prefer the longest coherent transcript between partial and final.
4. If final text is shorter than the latest partial and the partial starts with the final text, prefer the partial.
5. If text looks incomplete, wait briefly for continuation.
6. Never deliver the same utterance twice.
7. A callback from an old recognizer generation must never execute.
8. Do not restart the recognizer while an utterance finalization job is pending.
9. Cancel all stale jobs on stop, error, screen disposal, or new recognizer generation.

INCOMPLETE UTTERANCE HEURISTICS

Treat as probably incomplete:
- street word without a house number during destination input;
- sentence ending with words such as:
  "ir", "bet", "nes", "kad", "jeigu", "kai", "kur", "kaip", "kodėl";
- one very short content word;
- a transcript that changed within the last 700–1000 ms.

Treat as complete:
- destination with a number;
- known navigation command;
- question that has a complete interrogative structure;
- explicit command such as:
  "sustabdyk navigaciją"
  "kiek liko"
  "pakartok nurodymą"

Use different timing:
- known command: immediate after final result
- address with number: immediate after final result
- likely incomplete phrase: 1200–1800 ms grace
- normal casual sentence: 700–1200 ms grace

Do not add a long delay to every command.

PHASE 4 — NATURAL TURN-TAKING

Create explicit voice states if not already present:

IDLE
LISTENING
USER_SPEAKING
FINALIZING
THINKING
SPEAKING
RECOVERING

Rules:

- onBeginningOfSpeech → USER_SPEAKING
- onEndOfSpeech → FINALIZING, not immediately IDLE
- while FINALIZING, do not restart recognizer
- while THINKING, recognizer remains stopped
- while TTS SPEAKING, recognizer remains stopped unless real barge-in is intentionally supported
- after TTS completes, wait 300–600 ms before restarting listening
- do not produce visible IDLE flicker between continuous-session phases
- one state owner only; avoid multiple places writing conflicting state values

Do not introduce full-duplex listening in this task.

PHASE 5 — ERROR RECOVERY

Handle:
- ERROR_NO_MATCH
- ERROR_SPEECH_TIMEOUT
- ERROR_CLIENT
- ERROR_RECOGNIZER_BUSY
- ERROR_INSUFFICIENT_PERMISSIONS
- ERROR_NETWORK
- ERROR_SERVER

Rules:

- NO_MATCH / SPEECH_TIMEOUT:
  quietly restart in continuous mode after a small backoff
- RECOGNIZER_BUSY:
  destroy and recreate once, then retry
- CLIENT:
  do not enter a tight retry loop
- stale callbacks:
  ignore by generation ID
- never show "Kentas klauso" when the recognizer is actually stopped

Use bounded retry counts and exponential or staged backoff.
