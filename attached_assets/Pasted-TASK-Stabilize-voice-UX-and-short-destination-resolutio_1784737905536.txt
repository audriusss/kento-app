PHASE 2 — FIX SHORT DESTINATION RESOLUTION

Required behavior:

"Taikos 61"
→ attempt navigation destination resolution

"važiuojam į Taikos 61"
→ attempt navigation destination resolution

"Akropolis"
→ resolve as a place near the current region or location

"važiuojam į Akropolį"
→ resolve as a place

"Lidl"
→ resolve nearby relevant place

"degalinė"
→ resolve nearest or relevant nearby fuel station

Resolution priority should be explicit and deterministic:

1. Known voice command
2. Navigation phrase with extracted destination
3. Plain short destination candidate
4. Nearby category/place candidate
5. AI chat fallback only when the input is clearly conversational

Do not send plausible destination text to AI chat before destination resolution is attempted.

Implement a conservative destination-candidate classifier.

Examples that should be treated as destination candidates:

- street name plus number
- one or several words matching a place/business name
- common POI category
- phrases beginning with:
  - "važiuojam į"
  - "važiuok į"
  - "naviguok į"
  - "rask"
  - "rodyk kelią į"

Examples that should remain AI chat:

- "kas tu esi?"
- "papasakok anekdotą"
- "koks oras?"
- "ką manai apie..."
- "paaiškink..."

Preserve Lithuanian letters and normalize safely.

Do not remove address numbers.

Do not over-strip destination text.

Handle Lithuanian grammatical forms where practical, but do not invent a large NLP system.

If multiple places match:

- prefer proximity to current location
- prefer exact or strong name match
- ask a short clarification only when ambiguity is meaningful

Example:

"Radau kelis Akropolius. Kurį pasirinkti?"

If no result is found, say:

"Nepavyko rasti tikslo. Pasakyk miestą arba tikslesnį adresą."

PHASE 3 — HANDS-FREE CONVERSATION MODE

Current bad flow:

press microphone
→ user speaks
→ Kentas responds
→ user must press microphone again

Replace this with an explicit continuous driving conversation mode.

Required UX:

The user presses the microphone once to activate Kentas listening mode.

After activation:

- Kentas listens
- the user speaks
- command is processed
- Kentas speaks a response
- after TTS finishes, speech recognition starts again automatically
- the loop continues until the user explicitly stops listening

The microphone button should act as a toggle:

OFF
→ one press enables continuous listening mode

ON
→ one press disables continuous listening mode

Show clear UI state:

- "Kentas išjungtas"
- "Klausau..."
- "Atpažįstu..."
- "Kentas galvoja..."
- "Kentas kalba..."
- "Laukiu komandos..."

Do not restart speech recognition while TTS is still speaking.

Do not allow recognizer and TTS to capture each other.

Implement a proper state machine, not scattered booleans.

Suggested states:

Idle
Listening
Processing
Speaking
RestartDelay
Error

Create a single source of truth for voice session state.

For example:

voice/VoiceSessionState.kt
voice/VoiceSessionController.kt

Use StateFlow or an equivalent existing project pattern.

Required loop:

Idle
→ user enables mode
→ Listening
→ speech result
→ Processing
→ Speaking
→ TTS completion callback
→ short safe delay
→ Listening

Stop conditions:

- user taps microphone toggle off
- user says:
  - "Kentai, nustok klausyti"
  - "išjunk mikrofoną"
  - "baik klausytis"
- app goes to background
- permission is revoked
- unrecoverable recognizer error

Navigation stop must NOT automatically disable Kentas listening mode unless that is already intentional UX.

PHASE 4 — SPEECH RECOGNIZER ERROR RECOVERY

Handle common Android SpeechRecognizer errors without trapping the app in a broken state.

Examples:

- ERROR_NO_MATCH
- ERROR_SPEECH_TIMEOUT
- ERROR_NETWORK
- ERROR_RECOGNIZER_BUSY
- ERROR_CLIENT
- ERROR_INSUFFICIENT_PERMISSIONS

Expected behavior in continuous mode:

NO_MATCH or SPEECH_TIMEOUT
→ silently retry after a small delay

RECOGNIZER_BUSY
→ cleanly cancel/reset and retry once

NETWORK
→ announce a concise message only when useful, then allow retry

PERMISSION
→ stop listening mode and show actionable UI message

Prevent rapid infinite restart loops.

Use bounded retry/backoff.

PHASE 5 — TTS COORDINATION

Use TTS utterance progress callbacks.

The recognizer may restart only after onDone or a handled onError callback.

Do not estimate TTS completion using fixed timers alone.

A small post-TTS delay is allowed to prevent the recognizer from hearing the speaker output.

Example delay:

300–700 ms

Keep it configurable as a constant.

If TTS is interrupted, the voice session state must recover predictably.
