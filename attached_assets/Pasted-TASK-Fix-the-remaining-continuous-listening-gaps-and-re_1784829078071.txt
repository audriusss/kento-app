REQUIRED ARCHITECTURE

Create one single owner for the continuous listening loop.

Only one function may schedule the next recognizer start.

Suggested API:

requestListeningRestart(
    reason: RestartReason,
    delayMs: Long
)

It must:

1. Cancel the previous restart job.
2. Increment a restart token.
3. Refuse to start while:
   - TTS is speaking
   - state is THINKING
   - state is PROCESSING
   - state is FINALIZING
   - an utterance grace job is active
   - continuous mode is disabled
4. Start only if no recognizer session is currently active.
5. Ignore stale jobs by token/generation.
6. Never create two simultaneous SpeechRecognizer instances.

Remove or delegate all other direct restart scheduling.

VOICE LOOP STATES

Use one authoritative state machine:

IDLE
STARTING
LISTENING
USER_SPEAKING
FINALIZING
PROCESSING
THINKING
SPEAKING
RESTART_WAIT
ERROR

Rules:

- startListening requested → STARTING
- onReadyForSpeech → LISTENING
- onBeginningOfSpeech → USER_SPEAKING
- onEndOfSpeech → FINALIZING
- utterance delivered → PROCESSING
- AI request pending → THINKING
- TTS begins → SPEAKING
- TTS ends → RESTART_WAIT
- restart succeeds → STARTING → LISTENING

Never set IDLE between normal continuous-mode phases.

The UI must show “Kentas klauso” only in:

- LISTENING
- USER_SPEAKING

During STARTING or RESTART_WAIT show a neutral state such as:

“Kentas ruošiasi klausyti…”

Do not lie that the microphone is listening when it is not ready.

NO-MATCH / TIMEOUT LOOP

For ERROR_NO_MATCH and ERROR_SPEECH_TIMEOUT:

- do not enter IDLE;
- transition to RESTART_WAIT;
- restart once after 300–500 ms;
- reset retry count after onReadyForSpeech;
- do not announce an error to the user;
- do not stack restart jobs.

For ERROR_RECOGNIZER_BUSY:

- cancel pending restart;
- destroy the current recognizer once;
- recreate after 800 ms;
- prevent a destroy/recreate loop.

For ERROR_CLIENT:

- log the previous state and caller;
- retry only once after 1000 ms;
- do not loop continuously.
